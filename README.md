# NPDL 評量設計工作室

教師可依年級、科目、課程活動與 NPDL 子向度，產生課前診斷、課後遷移及四層進程判讀資料。Q1–Q4 固定形成「概念理解、行動應用、生活遷移」能力階梯。

## 課程發想工具

獨立入口為 `/course-ideation/`，內建地理、化學、生物、物理、數學、國文、英文與社會共 8 組可重複載入的測試範例，並提供課程發想與校準鷹架：

1. 輸入年級、學科、單元名稱、教學主題與 3–5 個核心關鍵字，由 AI 整理主題群及後續素養對齊訊號；預設以台灣高中約 30–40 人班級、50 分鐘課時、一般教室與少量共用裝置為可行性基準，超出現場條件的構想會縮小成校內、公開資料、模擬或低科技版本。物理、化學、生物與地科會分別取得一般高中可能具有的基礎實驗器材目錄，但所有器材均視為需由教師確認數量、狀態與安全規範，並附無器材時的等效替代方案。
2. 依年級與學科篩選內建的 108 課綱學習表現、學習內容，再由 AI 從受控候選 ID 採用條目；教師可調整，調整後必須重新校準。
3. 從受控 6Cs 子向度目錄推薦 1–2 項、顯示正式四級學習進程，並生成「知識基礎、素養子向度、四要素整合實踐」三層學習成果。知識基礎同時顯示課綱依據、學術預期成果、成功指標與評量證據。
4. 以逆向設計鎖定遷移目標、核心理解、核心問題與具穩定 ID 的成功指標；教師確認前不開放下游評量。
5. 建立課前與課後 Q1–Q4 的評量目的與預期證據、逐節形成性證據、真實總結任務及學科四級規準。真實總結任務預設由 3–5 人小組在正常課時內，以校內可接觸的受眾、低成本材料及一般設備完成；任何特殊設備、網路或外部合作都必須附低科技替代方案。課程工具不撰寫正式題目；正式 Q1–Q4 仍由評量工作室依既有文件格式產生。
6. 評量證據採「AI 草稿 → 教師逐項編修 → 確認鎖定」；每筆形成性證據都包含後續教學決策。解鎖重編會讓下游藍圖與 Canvas 提示詞過期。
7. 依總節數、分鐘、必要活動、設備場地、先備經驗與差異化需求，倒推出逐節藍圖；每節均連結成果、成功指標、證據、6Cs 與 NPDL 四要素，並確定性組裝該節的學習單設計 brief。
8. 通過覆蓋與順序檢查後，系統直接組裝一份包含全部節次的 Gemini Canvas 完整提示包。教師一次貼入 Gemini，產生整個單元的逐節教案，以及每節各一份分為「知識基礎」與「NPDL 子向度思考」的學生學習單與教師判讀指引，再於 Gemini 中自行修改；App 不提供逐節教案或學習單編輯，也不提供逐節提示詞。

NPDL 學習設計四要素固定為「學習夥伴關係、學習環境、數位利用、教學實踐」，不使用含義不同的 4E 教學循環。完成學習終點與證據系統後，可將年級、學科、課程名稱、課綱原文、學習成果、成功指標、證據工具與選定子向度一次性帶入評量工作室；`CourseForm` 與舊版交接資料格式維持相容，交接資料最長保留 24 小時，讀取後即清除。

第一版不呼叫 Gemini API 生成、儲存或編輯完整逐節教案，也不使用瀏覽器外掛或跨站自動貼上。教師複製提示詞後，以自己的 Google 帳號開啟 Gemini、選擇 Canvas 並貼上；開啟 Gemini 的連結不攜帶任何專案資料或 query parameter。若上游課綱、終點、證據或課程限制已變更，既有藍圖會標示需要重新校準，正式提示詞也會停用。

108 課綱資料在開發期由指定 NotebookLM 筆記協助核對，再整理為版本化的分科 JSON 與來源 manifest 隨網站建置。瀏覽器執行時不連線 NotebookLM；無法取得受控條目時，介面允許教師加入明確標示為「教師自訂／未由系統核對」的依據。

首次呼叫 AI 前，介面會顯示供應商、模型、用途及完整傳送內容。教師同意後會記錄在目前瀏覽器，後續不逐次詢問，並可隨時撤回；API Key 不包含在傳送內容中。

課程發想工具一律使用教師自備的 Gemini、OpenAI 或 Grok API Key，預設模型為 Gemini 2.5 Flash；API Key 以未加密文字保存在目前瀏覽器，設定視窗可清除目前供應商或全部金鑰。課程發想不提供 Puter 免費模型，但評量工作室原有的 Puter 選項維持不變。

## Google Forms

系統可依序建立課前、課後兩份 Google Form。每份表單包含：

- 完整共用情境。
- Q1–Q3 三題必填單選題。
- Q4「概念理解、行動應用、生活遷移」三題必填長答。
- 建立完成後的教師編輯連結與學生作答連結。

建立流程為「建立空表單 → 寫入題目 → 發布並開啟回應」。若中途失敗，重試只會接續未完成階段，不會重建表單或重複寫入題目。

部署管理者需在 GitHub Repository variable 設定：

```text
VITE_GOOGLE_OAUTH_CLIENT_ID
```

Google Cloud 的 OAuth Web Client 必須：

1. 啟用 Google Forms API。
2. 完成 OAuth 同意畫面。
3. 將 `https://evan645test.github.io` 加入「已授權的 JavaScript 來源」。

OAuth Client ID 是公開識別碼，可作為 GitHub Repository variable；不得上傳 Client secret、Google access token 或服務帳戶金鑰。部署完成後，一般使用者只需按下建立問卷、登入 Google 並確認授權。

## AI 模型

- OpenAI、Gemini、Grok：使用者自行提供 API Key，僅保存在目前瀏覽器。
- Puter 免費模型：僅評量工作室提供；不需要 API Key，使用者登入 Puter 後使用自己的免費額度。

## 本機開發

需求：Node.js 24。

```bash
npm ci
npm test
npm run build
npm run dev
```

本機若要測試 Google Forms，可複製 `.env.example` 為 `.env.local`，填入 OAuth Client ID；`.env.local` 已被 Git 忽略。
