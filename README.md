# NPDL Studio

教師可依年級、科目、課程活動與 NPDL 子向度，產生課前診斷、課後遷移及四層進程判讀資料。Q1–Q4 固定形成「概念理解、行動應用、生活遷移」能力階梯。

**教師使用說明書**：[docs/使用說明書.md](docs/使用說明書.md)｜**10 分鐘速查卡**：[docs/教師10分鐘速查卡.md](docs/教師10分鐘速查卡.md)

## 單一連續流程

根網址開啟合併版：已有／未有課程 → NPDL 構造 → 課前 → 藍圖 → **同頁課後與匯出**。頂部不再切換「課程設計／評量設計」；舊 `?workspace=assessment` 會忽略並導回根路徑。舊 `/course-ideation/` 仍自動轉到合併首頁。

**已移除**：不經課程構造、Sidebar 步驟 1→2→3 直接產整包的獨立評量路徑。無課前種子時不可產課後。

課程工作區內建地理、化學、生物、物理、數學、國文、英文與社會共 8 組可重複載入的測試範例，並提供課程發想與校準鷹架：

1. 輸入年級、學科、單元名稱、教學主題與 3–5 個核心關鍵字，由 AI 整理主題群及後續素養對齊訊號；預設以台灣高中約 30–40 人班級、50 分鐘課時、一般教室與少量共用裝置為可行性基準，超出現場條件的構想會縮小成校內、公開資料、模擬或低科技版本。物理、化學、生物與地科會分別取得一般高中可能具有的基礎實驗器材目錄，但所有器材均視為需由教師確認數量、狀態與安全規範，並附無器材時的等效替代方案。
2. 依年級與學科篩選內建的 108 課綱學習表現、學習內容；AI 初次校準會各推薦 5 項，先採用其中最相關的 2 項，並以完整卡片固定顯示。教師可從 5 項推薦或其餘分類清單中各選 2 項；改選時階段二與後續成果會保留為舊版本，重新校準後才更新。
3. 從受控 6Cs 子向度目錄推薦 1–2 項、顯示正式四級學習進程，並生成「知識基礎、素養子向度、四要素整合實踐」三層學習成果。知識基礎同時顯示課綱依據、學術預期成果、成功指標與評量證據。
4. 以逆向設計鎖定遷移目標、核心理解、核心問題與具穩定 ID 的成功指標；教師確認前不開放下游評量。
5. 建立課前與課後 Q1–Q4 的評量目的與預期證據、逐節形成性證據、真實總結任務及學科四級規準。真實總結任務預設由 3–5 人小組在正常課時內，以校內可接觸的受眾、低成本材料及一般設備完成；任何特殊設備、網路或外部合作都必須附低科技替代方案。課程工具不撰寫正式題目；正式 Q1–Q4 仍由評量工作室依既有文件格式產生。
6. 評量證據採「AI 草稿 → 教師逐項編修 → 確認鎖定」；每筆形成性證據都包含後續教學決策。解鎖重編會讓下游藍圖與 Canvas 提示詞過期。
7. 依總節數、分鐘、必要活動、設備場地、先備經驗與差異化需求，倒推出逐節藍圖；每節均連結成果、成功指標、證據、6Cs 與 NPDL 四要素，並確定性組裝該節的學習單設計 brief。
8. 通過覆蓋與順序檢查後，系統直接組裝一份包含全部節次的 Gemini Canvas 完整提示包。教師一次貼入 Gemini，產生整個單元的逐節教案，以及每節各一份分為「知識基礎」與「NPDL 子向度思考」的學生學習單與教師判讀指引，再於 Gemini 中自行修改；App 不提供逐節教案或學習單編輯，也不提供逐節提示詞。
9. 創意孵化、課綱／6Cs／學習終點、評量證據與單元節次藍圖中的每張內容卡，都有獨立「AI 修改這張卡」入口。AI 只能回傳該卡的結構化 patch，預覽並套用後仍會以完整 Schema、受控 ID、證據關聯、節數與時間限制重新驗證，不會連帶改寫同階段其他卡片。
10. 階段一可匯入一份文字型 PDF、DOCX 或 UTF-8 TXT 教案（上限 10 MB，送往 AI 的擷取文字上限 40,000 字元）。AI 先整理年級、科目、目標、活動、評量、資源、限制與差異化支持，只有教師勾選並確認的結構化內容才會進入後續課綱、證據與節次提示；原始檔、檔名及完整擷取文字不保存。

NPDL 學習設計四要素固定為「學習夥伴關係、學習環境、數位利用、教學實踐」，不使用含義不同的 4E 教學循環。完成學習終點、證據系統與課前種子後，可於同頁「課後與匯出」產生課後遷移、Forms 與證據包；`CourseForm`、舊版交接資料與既有草稿格式維持相容。

第一版不呼叫 Gemini API 生成、儲存或編輯完整逐節教案，也不使用瀏覽器外掛或跨站自動貼上。教師複製提示詞後，以自己的 Google 帳號開啟 Gemini、選擇 Canvas 並貼上；開啟 Gemini 的連結不攜帶任何專案資料或 query parameter。若上游課綱、終點、證據或課程限制已變更，既有藍圖會標示需要重新校準，正式提示詞也會停用。

108 課綱資料在開發期由國家教育研究院正式發布的領域／科目 ODT 原檔抽取與核對，再整理為版本化的分科 JSON 與來源 manifest 隨網站建置；同碼原文仍有衝突的條目會排除並記錄。指定 NotebookLM 僅供開發期交叉查詢，瀏覽器執行時不連線 NotebookLM；無法取得受控條目時，介面允許教師加入明確標示為「教師自訂／未由系統核對」的依據。

首次呼叫 AI 前，介面會顯示供應商、模型、用途及完整傳送內容，包含 AI 修改時所需的現有產出與教師修改要求。教師同意後會記錄在目前瀏覽器，後續不逐次詢問，並可隨時撤回；API Key 不包含在傳送內容中。

教師自備的 Gemini、OpenAI 或 Grok 模型與 API Key（BYOK）在整個流程共用，預設模型為 Gemini 2.5 Flash；API Key 以未加密文字保存在目前瀏覽器，設定視窗可清除目前供應商或全部金鑰。合併版介面不提供 Puter 免費模型；舊 Puter 選擇會自動遷移至有效的 BYOK 模型或 Gemini 2.5 Flash，既有評量文件仍可讀取。

## Google Forms

系統可依序建立課前、課後兩份 Google Form。每份表單包含：

- 完整共用情境。
- Q1–Q3 三題必填單選題。
- Q4「概念理解、行動應用、生活遷移」三題必填長答。
- 建立完成後的教師編輯連結與學生作答連結。

建立流程為「建立空表單 → 寫入題目 → 發布並開啟回應」。若中途失敗，重試只會接續未完成階段，不會重建表單或重複寫入題目。

部署管理者需在 GitHub Repository variable 設定：

```text
VITE_GOOGLE_OAUTH_CLIENT_ID
```

Google Cloud 的 OAuth Web Client 必須：

1. 啟用 Google Forms API。
2. 完成 OAuth 同意畫面。
3. 將 `https://evan645test.github.io` 加入「已授權的 JavaScript 來源」；本機預覽另加 `http://127.0.0.1:4180` 與 `http://localhost:4180`。

完整步驟見 [docs/google-forms-oauth-setup.md](docs/google-forms-oauth-setup.md)。

OAuth Client ID 是公開識別碼，可作為 GitHub Repository variable；不得上傳 Client secret、Google access token 或服務帳戶金鑰。部署完成後，一般使用者只需按下建立問卷、登入 Google 並確認授權。

## AI 模型

- OpenAI、Gemini、Grok：全流程共用同一模型選擇；使用者自行提供 API Key，且僅保存在目前瀏覽器。
- Puter：合併版介面已移除；底層相容程式暫時保留供回滾使用。

## 本機開發

需求：Node.js 24。

```bash
npm ci
npm test
npm run build
npm run dev
```

更新受控課綱快照時執行 `npm run curriculum:build`。腳本會下載並快取國教院正式 ODT、排除無法消解的同碼原文衝突，再重建 `entries.json` 與 `manifest.json`；`.cache/` 不進入版本控制。

本機若要測試 Google Forms，可複製 `.env.example` 為 `.env.local`，填入 OAuth Client ID；`.env.local` 已被 Git 忽略。
